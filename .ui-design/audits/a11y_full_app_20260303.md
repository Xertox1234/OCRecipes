# Accessibility Audit Report

**Audit ID:** a11y_full_app_20260303
**Date:** 2026-03-03
**Target:** Entire application (client/screens + client/components + client/camera)
**WCAG Level:** AA
**Standard:** WCAG 2.1
**Files Audited:** 86 (37 screens, 48 components, 1 camera)

## Executive Summary

**Compliance Status:** Needs Improvement

| Severity  | Count   | % of Issues |
| --------- | ------- | ----------- |
| Critical  | 18      | 13%         |
| Serious   | 42      | 31%         |
| Moderate  | 42      | 31%         |
| Minor     | 33      | 25%         |
| **Total** | **135** | 100%        |

**Strengths:** The codebase demonstrates solid accessibility foundations -- most interactive elements have `accessibilityRole` and `accessibilityLabel`, the `Button` and `TextInput` components have well-structured a11y props, animations broadly check `reducedMotion`, and the `InlineError` component pattern is well-adopted. `accessibilityViewIsModal` is used on most modals.

**Key Gaps:** Missing image alt text, undersized touch targets (36-40px vs 44px minimum), unannounced dynamic status messages, missing progressbar roles on visual bars, and a few modals/animations missing a11y treatment.

## Recommendations

### Quick Wins (< 1 hour each)

1. **Add `accessible={false}` to all decorative images** (C1-C4, S19-S22) -- 11 issues, one-line fix each
2. **Add `accessibilityLabel` to content images** (C5-C7) -- 3 issues, one-line fix each
3. **Add `accessibilityLabel` to unlabeled TextInputs** (C8-C10) -- 3 issues, one-line fix each
4. **Add `accessibilityViewIsModal`** to FollowUpModal and GLP1 modal (C11-C12) -- 2 issues, one-line fix each
5. **Add `importantForAccessibility="no"`** to decorative icons (m1-m12) -- 12 issues, one-line fix each
6. **Add `accessibilityRole="alert"`** to error banners (S37-S38) -- 2 issues, one-line fix each

### Medium Effort (1-4 hours each)

1. **Increase all undersized touch targets to 44px** (S1-S18, M1-M8) -- 26 issues across ~20 files
2. **Add `accessibilityLiveRegion` and `announceForAccessibility`** to dynamic status messages (M9-M18) -- 10 issues
3. **Add progressbar role/value** to CalorieBudgetBar, MicronutrientBar (C14-C15) -- 2 component fixes
4. **Fix radio group semantics** (M26-M29) -- 4 screens need radiogroup wrapper
5. **Replace raw `<Text>` with `<ThemedText>`** in GLP1CompanionScreen and MenuScanResultScreen (C17, S29)

### Significant Effort (> 4 hours)

1. **Add accessible chart summaries** (C13, S27) -- WeightChart and FastingScreen SVG charts need computed text descriptions
2. **Add accessibilityActions for long-press alternatives** (S41-S42) -- Needs swipe-to-delete or action menus
3. **Add `accessibilityRole="header"` to all section headings** (M21-M22) -- Systematic pass across all screens
4. **Fix Chip component to pass through accessibilityRole** (M30) -- Component change affecting all consumers

### Patterns to Establish

1. **Image a11y pattern**: All `<Image>` components should either have `accessibilityLabel` (content images) or `accessible={false}` (decorative). Add a lint rule.
2. **Progress bar pattern**: All visual progress bars should have `accessibilityRole="progressbar"` with `accessibilityValue`.
3. **Status message pattern**: All dynamically appearing messages should use `accessibilityLiveRegion` (Android) + `announceForAccessibility` (iOS).
4. **Touch target pattern**: All interactive elements should have `minHeight: 44` or equivalent hitSlop. Add a lint rule.
5. **Radio group pattern**: Mutually-exclusive selections should use `accessibilityRole="radio"` inside a `radiogroup` container.
