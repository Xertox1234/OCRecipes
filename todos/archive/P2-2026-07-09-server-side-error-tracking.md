# Add server-side error tracking for production 5xx errors

## Summary

The Express backend has no error-tracking/APM integration. Every unhandled exception and 5xx
response is logged via pino to stdout only ([server/index.ts:174](../server/index.ts#L174),
`logger.error({ err }, "internal server error")`) and relies entirely on Railway's raw log
viewer for visibility — no alerting, aggregation, dedup, or stack-trace grouping.

## Background

Surfaced during a logging-architecture review (2026-07-09). `@sentry/react-native` is already
wired client-side in `client/lib/reporter.ts` (DSN from `EXPO_PUBLIC_SENTRY_DSN`, active outside
`__DEV__`), but there is no equivalent on the server — `@sentry/node` (or similar) is not a
dependency. Today, a production incident that manifests as both a client crash and a server 500
shows up in two disconnected places (Sentry for the client, Railway stdout for the server), with
no shared trace/request ID linking them — the `requestId` pino already generates per request
(via `pino-http` + AsyncLocalStorage, see [server/lib/logger.ts](../server/lib/logger.ts) and
[server/index.ts:128-162](../server/index.ts#L128-L162)) never reaches the client's error report.

## Acceptance Criteria

- [ ] Add a server-side error-tracking integration (Sentry Node SDK is the natural choice given
      the existing Sentry project/DSN pattern on the client, but confirm before committing —
      could also be a Railway-native or other APM option).
- [ ] Wire it into the global Express error handler ([server/index.ts:164-187](../server/index.ts#L164-L187))
      and the `uncaughtException`/`unhandledRejection` handlers (`server/index.ts:27`, `:34`).
- [ ] Attach the per-request `requestId` (already generated by `pino-http`/ALS) as tag/context on
      reported events, so a server error and the corresponding client-side report (if any) can be
      correlated.
- [ ] Ensure no PII/secrets leak into reported events — mirror the client's `sendDefaultPii: false` + explicit `Authorization`-header scrub pattern from `client/lib/reporter.ts`.
- [ ] Document the new env var(s) (e.g. a server-side Sentry DSN) in CLAUDE.md's Environment
      Variables section.

## Implementation Notes

- Keep this server-only — do not touch the existing client Sentry wiring.
- `server/lib/logger.ts`'s pino `mixin()` already has access to `requestId`/`userId` via
  AsyncLocalStorage (`server/lib/request-context.ts`) — the new integration should read from the
  same context rather than re-deriving request identity.
- No `redact` config exists on the pino logger today; this todo is scoped to error-tracking only,
  not a general logging-redaction hardening pass (that would be a separate, narrower todo if
  pursued).

## Dependencies

- None — additive change, no coupling to other in-flight todos.

## Risks

- New third-party network calls from the server on every unhandled error — must not block or
  slow the error-response path (fire-and-forget / async capture only).
- Cost: most APM/error-tracking services bill per event; needs a sane sampling/rate-limit default
  to avoid surprise usage on a noisy error loop.
