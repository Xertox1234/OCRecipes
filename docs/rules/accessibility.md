# Accessibility Rules

- All `fullScreenModal` and `modal` screens must have `accessibilityViewIsModal={true}` on the root container — VoiceOver navigates behind the modal without it
- Decorative icons inside labeled Pressables must have `accessible={false}` — VoiceOver double-focuses on icon + container otherwise
- `Pressable` with `disabled={true}` auto-propagates to `accessibilityState.disabled` on RN 0.81 — an explicit `accessibilityState={{ disabled: true }}` is only required on NON-Pressable touchables (TouchableOpacity, custom Views), where the prop does not propagate
- Error messages must use `accessibilityLiveRegion="assertive"` not `"polite"` — use the `InlineError` component, not `Alert.alert()`
- Radio buttons: use `accessibilityState={{ selected: bool }}` not `{{ checked: bool }}` — `checked` maps to checkbox semantics on TalkBack; radio chip rows need a `role="radiogroup"` wrapper `View`
- iOS Fabric (`newArchEnabled: true`, this app) drops the _wrapper_ announcement for `radiogroup`, `tablist`, and other ARIA group roles — its `accessibilityValue` mapping (`RCTViewComponentView.mm`) special-cases only `checkbox`/`radio`; the legacy Paper mapping that spoke the group role explicitly does not run on Fabric. Android is unaffected (`ReactAccessibilityDelegate.java` maps group roles via `setRoleDescription`). **Decision (2026-07-19): accept-and-document, do not mitigate** — VoiceOver already announces each item as "radio button" (RN's own source comment: screen-reader users are assumed already familiar with radio/checkbox controls from the web, which is why RN speaks the control name at all), so the per-item role still gives users the standard affordance to infer single-select from even without a spoken group cue; a supplemental iOS-only cue can't be verified without a live device pass and risks double-announcing. Keep using the group-role wrapper (structure + Android + forward-compat); do not add per-screen compensating cues for the iOS gap. See `docs/solutions/design-patterns/radio-checkbox-group-container-pattern-2026-05-13.md`.
- Progress bars need `accessibilityRole="progressbar"` + `accessibilityValue={{ min: 0, max: 100, now: value }}`
- Decorative emoji must be wrapped in a `Text` with `accessible={false}` — VoiceOver announces them literally otherwise
- Badges that are purely decorative inside a parent with an `accessibilityLabel` need `accessible={false}` — prevents double-announcement
- Never set `accessible={true}` on a banner/card wrapper that contains an interactive child (dismiss/CTA Pressable) — VoiceOver/TalkBack collapses children into one node and the button becomes unreachable. Put the role/label on the text node and let the Pressable stay its own a11y node
- Never apply `accessibilityRole="checkbox"` (or `"radio"`, `"switch"`) to a non-`Pressable` element — assistive tech announces the toggle affordance but the gesture does nothing. For visual-only status indicators, set `accessible={false}` + `importantForAccessibility="no"` and roll the state into a parent group's `accessibilityLabel`
- `pointerEvents="none"`, `opacity: 0`, and `height: 0`+`overflow: hidden` do NOT remove a mounted view from the a11y tree — any visually-hidden-but-mounted surface must also set `accessibilityElementsHidden={hidden}` + `importantForAccessibility={hidden ? "no-hide-descendants" : "auto"}`. When two surfaces swap (expanded ⇄ collapsed), hide whichever is currently invisible — BOTH directions — and resync when `reducedMotion` changes visibility at runtime. See `docs/solutions/conventions/a11y-hide-visually-hidden-surfaces-2026-06-10.md`

Announcements — iOS imperative announce vs Android live region (the double-announce family):

- Traits alone NEVER announce on iOS: `accessibilityLiveRegion` is Android-only, and neither `accessibilityRole="alert"` nor `accessibilityState={{ busy: true }}` posts a VoiceOver announcement. Every async transition (success, error, limit-reached, pending) needs `AccessibilityInfo.announceForAccessibility` for iOS coverage — never remove it thinking the alert role or busy trait covers iOS
- Never double-announce a platform: gate the imperative announce to `Platform.OS === "ios"` ONLY when the element still has an Android live region covering the same change; when there is no live region, announce on BOTH platforms ungated
- `InlineError` fires its own iOS-gated announce internally — never add a second announce in the `onError` handler that renders it (iOS hears it twice)
- Announce mutation errors from the `onError` handler (fires once per failure), not a `useEffect` keyed on `isError` (re-fires on stale/unrelated re-renders); announce pending/busy states on the idle→busy edge with a prev-value ref guard (silent on mount and on clear)
- A `"polite"` live region on a SHARED container that wraps a structural swap (e.g. `Text`↔`ActivityIndicator`) or a descendant's state change re-reads the ENTIRE subtree on every descendant change (TalkBack composes `CONTENT_CHANGE_TYPE_SUBTREE`). When one container is the sole announcer for multiple states/variants, drop the container live region and announce imperatively per transition instead
- Key imperative announces on the meaningful CONTENT that changed (an async-loaded name/value), not only the variant/discriminator string — content that mutates while the discriminator stays the same silently stops announcing where the dropped live region used to cover it; add a second edge-guarded effect keyed on that value. Precedent: `client/camera/components/ProductChip.tsx` (7 variants + the smart-confirm busy swap share one imperative announcer; a `productName`-keyed effect covers the in-place BARCODE_LOCKED→PRODUCT_LOADED update)
