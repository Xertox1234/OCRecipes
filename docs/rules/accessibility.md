# Accessibility Rules

- All `fullScreenModal` and `modal` screens must have `accessibilityViewIsModal={true}` on the root container — VoiceOver navigates behind the modal without it
- Decorative icons inside labeled Pressables must have `accessible={false}` — VoiceOver double-focuses on icon + container otherwise
- `Pressable` with `disabled={true}` auto-propagates to `accessibilityState.disabled` on RN 0.81 (verified in the Pressable source) — an explicit `accessibilityState={{ disabled: true }}` is only required on NON-Pressable touchables (TouchableOpacity, custom Views), where the prop does not propagate
- Error messages must use `accessibilityLiveRegion="assertive"` not `"polite"` — use `InlineError` component, not `Alert.alert()`
- Async state transitions (success, error, limit-reached) must call `AccessibilityInfo.announceForAccessibility` on iOS — **exception**: when using `InlineError` for the error display, do NOT also call `announceForAccessibility` in the `onError` handler; `InlineError` already fires its own iOS-gated announce internally, so adding a second call causes double-announce on iOS
- Radio buttons: use `accessibilityState={{ selected: bool }}` not `{{ checked: bool }}` — `checked` maps to checkbox semantics on TalkBack
- Radio chip rows need a `role="radiogroup"` wrapper `View`
- Progress bars need `accessibilityRole="progressbar"` + `accessibilityValue={{ min: 0, max: 100, now: value }}`
- Decorative emoji must be wrapped in a `Text` with `accessible={false}` — VoiceOver announces them literally otherwise
- `accessibilityLiveRegion` is Android-only — always pair with `AccessibilityInfo.announceForAccessibility()` for iOS coverage, but gate the announce call to `Platform.OS === "ios"` when the same element already has a live region (Android `TYPE_ANNOUNCEMENT` + live region = double-announce). **Exception** — a single `accessibilityLiveRegion="polite"` on a _shared container_ that wraps a structural swap (e.g. `Text`↔`ActivityIndicator`) or a descendant's `accessibilityState` change re-reads the ENTIRE subtree on every descendant change, not just the intended cue (TalkBack composes a `CONTENT_CHANGE_TYPE_SUBTREE` from the container). When one container is the sole announcer for multiple states/variants, drop the container live region and announce imperatively per transition on BOTH platforms instead (no iOS gate — there's no live region left to double-announce against). **Key the imperative announces on the meaningful CONTENT that changed (an async-loaded name/value), not only the variant/discriminator string** — a live region re-reads on _any_ descendant change, so if you replace it with a discriminator-keyed effect, content that mutates while the discriminator stays the same (e.g. an async product name loading into an already-shown card) silently stops announcing on Android, where the dropped live region used to cover it. Use a second edge-guarded effect keyed on that value. Precedent: `client/camera/components/ProductChip.tsx` (7 variants + a smart-confirm busy swap shared one polite container; a `productName`-keyed effect covers the async BARCODE_LOCKED→PRODUCT_LOADED in-place update)
- `accessibilityRole="alert"` does NOT auto-announce on iOS either — it only affects focus/traits, it does not post a `UIAccessibilityAnnouncementNotification`. An error banner with `role="alert"` + `accessibilityLiveRegion="assertive"` announces on Android (live region) but is silent on iOS. Never remove the iOS-only `announceForAccessibility` thinking the alert role covers iOS — it doesn't. For mutation errors, put the iOS-gated announce inside the `onError` handler (fires once per failure) rather than a `useEffect` keyed on `isError` (re-fires on stale/unrelated re-renders). **Exception**: `InlineError` already contains a `Platform.OS === "ios"` gated `announceForAccessibility` internally — do not add a second announce in the `onError` handler when `InlineError` displays the error, or both fire and iOS hears it twice
- Badges that are purely decorative inside a parent with an `accessibilityLabel` need `accessible={false}` — prevents double-announcement
- Never set `accessible={true}` on a banner/card wrapper that contains an interactive child (dismiss/CTA Pressable) — VoiceOver/TalkBack collapses children into one node and the button becomes unreachable. Put `accessibilityRole`/`accessibilityLabel` on the text node and let the Pressable stay its own a11y node
- Never apply `accessibilityRole="checkbox"` (or `"radio"`, `"switch"`) to a non-`Pressable` element — assistive tech announces the toggle affordance but the gesture does nothing. For visual-only status indicators, set `accessible={false}` + `importantForAccessibility="no"` and roll the state into a parent group's `accessibilityLabel` (e.g., "Accepted commitment: …" vs "Commitment: …")
- `accessibilityState={{ busy: true }}` does NOT post a VoiceOver announcement on iOS — it only sets a trait (same principle as `role="alert"` above). A button that enters a pending/loading state (spinner swap + `busy`) is therefore silent for iOS VoiceOver users while the async work runs. Pair the `busy` flag with an `AccessibilityInfo.announceForAccessibility("…")` fired once on the idle→busy edge (guard with a prev-value ref so it doesn't fire on mount or on clear). Gate to `Platform.OS === "ios"` **only when** the element/container still has an `accessibilityLiveRegion` (Android is covered by the live region — a second announce double-speaks); if there is no live region, announce on BOTH platforms with no gate. Precedent: `client/camera/components/ProductChip.tsx` smart-confirm pending state announces `"Analyzing photo…"` on both platforms — it deliberately has **no** container live region (a shared polite region re-read the whole chip on the busy swap; see the `accessibilityLiveRegion` rule above)
- `pointerEvents="none"`, `opacity: 0`, and `height: 0`+`overflow: hidden` do NOT remove a mounted view from the a11y tree — any visually-hidden-but-mounted surface must also set `accessibilityElementsHidden={hidden}` + `importantForAccessibility={hidden ? "no-hide-descendants" : "auto"}`. When two surfaces swap (expanded header ⇄ collapsed bar), hide whichever is currently invisible — BOTH directions. If `reducedMotion` forces a surface permanently invisible, its visibility state must agree AND resync when the setting toggles at runtime. See `docs/solutions/conventions/a11y-hide-visually-hidden-surfaces-2026-06-10.md`
