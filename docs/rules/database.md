# Database Rules

- Never use `onConflictDoNothing({ target })` with partial unique indexes ‚Äî omit the target arg entirely (PG rejects at runtime; causes live test failures)
- Use `onConflictDoUpdate` for cache tables, not `onConflictDoNothing` ‚Äî the latter silently skips expired-entry updates, causing `!`-assertion crashes on the stale row
- Always pair `.default([])` with `.notNull()` on array columns ‚Äî `.default([])` alone keeps the TS type `T[] | null` and crashes on legacy NULLs
- EXCEPTION ‚Äî safety/correctness-critical derived columns must be NULLABLE with no default (not `NOT NULL DEFAULT []`): `[]` cannot distinguish "derived, genuinely empty" from "not yet derived", so a `[]` default fails OPEN between migration and backfill. Use `null` = "not derived" (predicate treats it as conservatively unsafe); `[]` = "derived, empty" = safe. Write paths must always store a concrete array. See `docs/solutions/conventions/nullable-not-empty-for-derived-safety-columns-2026-05-17.md`
- Derive the `Insert` type for a table with a NULLABLE typed `jsonb` column from `typeof table.$inferInsert` ‚Äî NOT `z.infer<typeof createInsertSchema(table)>`: drizzle-zod 0.7.x loosens a nullable `.$type<>()` jsonb column to a generic `Json` union, breaking `.values()` insert type-safety
- Polymorphic FK always requires a discriminator column (e.g., `recipeType`) alongside the FK ‚Äî never a bare `recipeId` without type context
- Never store large blobs (images, receipts > 1 KB) in DB columns ‚Äî use file/object storage (Cloudflare R2)
- Multi-phase background jobs: design the eligibility query to catch phase-1-complete + phase-2-incomplete as a retriable state, not a dead end
- Always use `Promise.all` for parallel queries inside transactions ‚Äî never sequential `await` (causes N sequential round-trips)
- Never re-query after an insert to build the response ‚Äî construct from insert params + returned id in-memory
- Polymorphic batch fetch: collect IDs first, batch with `.inArray()`, resolve with `Map` lookup ‚Äî never loop-query
- Replace-by-DELETE-then-INSERT: dedupe the input array with `[...new Set(ids)]` before insert when removing `onConflictDoNothing`; DELETE alone does not prevent duplicate-row failures on a (userId, x) unique constraint when caller-supplied ids contain repeats
- Never accept a filter parameter in a storage function's type signature that the SQL query does not actually consume ‚Äî silently dropped filters cause downstream personalization/label bugs (callers expect filtering or boosting that never happens)
- Never interpolate a JS array into a raw `sql` template inside `ANY()`/`IN (...)` ‚Äî `sql\`col = ANY(${arr})\``binds the array as a scalar and PG throws "malformed array literal" at runtime; use`inArray(col, arr)`from`drizzle-orm`
- `CURRENT_TIMESTAMP` and `now()` return transaction-start time ‚Äî for tests that need distinct `createdAt` ordering within one transaction, pass explicit `new Date(baseTime - N)` values; do not rely on `setTimeout` between inserts
- `TIMESTAMP WITHOUT TIME ZONE` columns drop offset info on PG round-trip ‚Äî storage-layer recency assertions cannot use tight `before <= stamp <= after` windows; use a 24h-tolerant `Math.abs(stamp - Date.now()) < 24*60*60*1000` check to catch hardcoded-epoch regressions while surviving max 14h TZ drift. Tight request-window checks belong in route tests where the Date is not PG-roundtripped.
- Never persist a caller-computed aggregate (count, level, averaged value) derived from a pre-write read ‚Äî under concurrency two requests compute the same aggregate from a stale snapshot and lost-update each other. Take an advisory lock as the first statement in the transaction, then RECOMPUTE the aggregate from the source rows AFTER the mutating insert and return it. A lock alone is insufficient ‚Äî it serializes the writes but still persists the stale value. See `docs/solutions/design-patterns/recompute-aggregate-under-lock-2026-05-17.md`
- The same race applies to per-row derived values (a boolean/classification stored ON the row being inserted, computed by comparing the new row against existing sibling rows) ‚Äî never compute it from a pre-transaction read. Move the comparison inside the same advisory-locked transaction: SELECT the comparison-candidate rows under the lock BEFORE the insert, compute the per-row value from that snapshot, insert with it. The storage function drops the caller-supplied per-row parameter and returns the computed value. See `docs/solutions/design-patterns/recompute-aggregate-under-lock-2026-05-17.md` "Per-row derived values have the same race"
- Never cast `db.execute(sql\`...\`)`results via`as T`or`execute<T>(...)`alone ‚Äî both are compile-time-only assertions on`Record<string, unknown>[]`rows; Drizzle provides no runtime type safety for raw SQL results. Use Zod parse of`result.rows[0]` against the table's inferred select schema to catch schema drift at runtime.
- A one-shot prod-ops script that does a destructive/irreversible DB write (backfill, mass `UPDATE`, column drop) must gate the write on an explicit opt-in CLI flag (`--allow-prod-backfill`, like `seed-recipes.ts`'s `--allow-prod-seed`) ‚Äî NEVER on `process.env.NODE_ENV === "production"`: these run via `railway run`, which does not set `NODE_ENV`, so an env-keyed guard never fires and the write runs unguarded. Make idempotency the WHERE clause (`WHERE col = false`), report `.returning().length` as the affected count, and `isMain`-guard `main()` so tests can import the helpers. See `docs/solutions/conventions/prod-ops-script-guard-on-flag-not-node-env-2026-06-20.md`
- Every caller of a day-bucketed read (`getDailySummary`, `getDailyLogs`, anything built on `getDayBounds`) must pass a timezone ‚Äî never let the `tz = "UTC"` default apply on a user-specific surface. Request paths: `parseTimezone(req.headers["x-timezone"])` threaded through the service (client sends `X-Timezone: getDeviceTimezone()` AND keys the query cache by tz). Background jobs: the stored `users.timezone`, resolved BEFORE the first day-bucketed call. For "today", omit the date param and bucket the now-instant in tz ‚Äî a `?date=YYYY-MM-DD` string parses as a UTC-midnight instant and buckets the previous local day for UTC-negative users. See `docs/solutions/conventions/timezone-day-bucketing-threading-2026-06-10.md`
