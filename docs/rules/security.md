# Security Rules

> Binding directives only. Full rationale, failure-mode walkthroughs, and precedent paths live in the local knowledge base (`docs/solutions/` security-tagged — gitignored, auto-surfaced by the pattern-injection hook; see `security-rules-extended-rationale-2026-06-05.md`) and in this file's git history.

- Every resource lookup must scope by `userId` AND visibility (`eq(t.isPublic, true)` OR `eq(t.authorId, userId)`) — reads, not just mutations.
- The `isPublic`/ownership guard must appear on BOTH the read AND write path of any storage fn that accepts user-supplied IDs (an unfiltered write binds a private resource to a user and leaks its metadata on read).
- Polymorphic-FK junction inserts (`(parent_id, target_id, target_type)`) must verify the TARGET's ownership/visibility with its own `EXISTS` guard — a parent-only guard scopes the wrong row. The resolve path must hide non-visible-but-existing targets without orphan-deleting them.
- A single-resource read taking a user-supplied id must take `userId` as a REQUIRED param and filter `id = ? AND (isPublic OR authorId = userId)` in SQL (defense-in-depth; keep the route check too). `authorId = NULL` is always false in SQL, so an empty `userId` fails safe to public-only.
- A batch id→metadata resolver must scope by the relationship/junction table that AUTHORIZES the caller (`INNER JOIN ... ON ... AND userId = ?`), NEVER by the resource's own `isPublic` flag (which both drops entitled-but-now-private rows and is not an IDOR guard). The junction must be append-only. Precedent: `getCommunityRecipeTitlesByIds` via `recipe_dismissals` (`server/storage/community-recipes.ts`).
- A cross-user resource must return the SAME response as a missing one — never a distinguishable `403`. Collapse existence + ownership into one guard (`if (!x || x.userId !== req.userId)`); where the resource is optional data, resolve a cross-user hit to `undefined` and proceed as if absent.
- Storage update fns must accept an explicit field whitelist — never `Partial<User>` or a spread of arbitrary input (mass-assignment).
- All route request bodies must be Zod-validated before any field access (`schema.safeParse(req.body)`) — never `req.body.x` raw.
- `req.userId` is a UUID string — never `parseInt` it (returns `NaN`, silently bypasses ownership checks).
- Sanitize ALL prompt roles (`user`, `assistant`, `system`) before sending to OpenAI — never only `user`. Never trust parameters that "look server-generated" in AI prompt inputs — sanitize at the prompt boundary.
- Rate-limit every AI/OpenAI endpoint — apply a limiter from `server/middleware/rate-limiter.ts` before the handler.
- Safety/classifier regexes consuming across newlines must bound every `.*`/`[\s\S]*` gap. For a safety/injection detector the bound VALUE must be ≥ the upstream input-length cap (e.g. `[\s\S]{0,2000}` against a 2000-char limit) — a smaller bound is itself a bypass. Routing-only heuristics may use a tighter bound.
- Premium-gate BOTH the read AND write endpoints of a premium feature — gating only writes leaves data readable for free.
- Never index `TIER_FEATURES[tier]` with the raw stored `users.subscriptionTier` (not reset on expiry → lapsed users keep paid features). Use `storage.getEffectiveTierForUser(userId)` (`server/storage/users.ts`) for route gates and storage limit checks. Fallback: `resolveEffectiveTier(tier, expiresAt)` from `@shared/types/premium` when you already hold a subscription record. Do NOT route per-request gates through the cached `resolveSubscriptionTierFeatures` (pulls `getUserVerificationStats` + 60s cache; breaks route tests). EXEMPTION: B2B `ApiTier` has no expiry — never pass it through `resolveEffectiveTier`/`getEffectiveTierForUser`.
- Legal attestation fields (age confirmation, ToS, marketing opt-in) must propagate from the UI checkbox through every intermediate hook to the API call — never hardcode `true` in an auth hook/wrapper (the server gate still enforces, but a hardcoded client bypass falsifies the legal record).
- Stamp consent/audit timestamps server-side from `new Date()` INSIDE the storage fn, gated by a boolean `recordConsent`-style flag — never accept the timestamp as a parameter, even at the storage layer; clients send a boolean intent, never a timestamp. Omit the column from the Zod input schema, and defensively strip it from caller spreads (`const { healthDataConsentAt: _strip, ...safe } = updates`).
- Consent/audit timestamps are append-only — `COALESCE(existing, incoming)` in partial updates, existence guards in transactional upserts, so a re-stamp can't overwrite the original.
- `pull_request_target` workflows with repo secrets must execute only trusted base-branch code — never checkout/source/import/install/execute PR-head files; fetch PR head only as diff data.
- Any deployment behind a reverse proxy must set `app.set("trust proxy", <hop count>)` — numeric, NEVER `true` (leftmost-XFF is client-spoofable) — and re-verify every `req.ip` consumer (rate-limit keying) on any topology change; custom keyGenerators suppress express-rate-limit's misconfig warnings, so the collapse-to-one-bucket failure is silent. Trust client-IP headers (X-Real-IP / CF-Connecting-IP) only when gated on the platform that overwrites them (e.g. `RAILWAY_ENVIRONMENT_NAME`).
- Never derive public-bucket/CDN object keys from user identifiers — an unsalted hash of an exposed id (e.g. `sha256(userId)`, with `authorId` public via community recipes) is still forward-computable. Runtime uploads: random keys (`crypto.randomUUID()`). Idempotent migrations: mix the content hash in (`sha256(userId + ":" + sha256(bytes))`) or HMAC with a server secret. See `docs/solutions/logic-errors/public-cdn-keys-derivable-from-exposed-ids-2026-06-10.md`.
