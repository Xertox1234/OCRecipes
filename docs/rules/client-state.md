# Client State Rules

- Always use `apiRequest()` from `@/lib/query-client` for all server communication — never raw `fetch()` in components or hooks
- Always use the `Authorization` header for auth tokens — never cookies
- After an irreversible server mutation (account delete, payment, hard-delete), wrap local cleanup (`tokenStorage.clear`, `AsyncStorage.removeItem`) in try/catch and never throw — a post-success error makes the user "retry" a destructive action that already succeeded
- **`apiRequest` error contract**: `apiRequest` runs `throwIfResNotOk`, so it throws `ApiError("<status>: <serverBody>", code)` on any non-ok response. Therefore: (a) it ALWAYS throws on non-ok — an `if (!res.ok) { … }` guard after `await apiRequest(...)` is unreachable dead code that silently kills a branch keyed on a string it was the sole producer of. See `docs/solutions/logic-errors/dead-apirequest-guard-hides-broken-error-branch-2026-05-31.md`. (b) `error.message` is the raw server body (prefixed `"404: …"`) — never render it or pass it to `AccessibilityInfo.announceForAccessibility`; show static user-safe copy. See `docs/solutions/conventions/apierror-code-driven-static-copy-2026-05-31.md`. (c) branch on `error instanceof ApiError && error.code === "<CODE>"`, never on `error.message` (prefixed, so `=== "404"` never matches); to give a status distinct UX, return a _specific_ `ErrorCode` server-side, never the generic `VALIDATION_ERROR`
- The centralized 401→logout interceptor fires the session-expiry signal ONLY when the 401 body carries a session-token code (`TOKEN_EXPIRED`/`TOKEN_INVALID`/`TOKEN_REVOKED`, emitted only by `server/middleware/auth.ts`) — never on a bare token-bearing 401. A route handler's own 401 (e.g. the `DELETE /api/auth/account` password check) returns `UNAUTHORIZED` and must NOT log the user out. Expiry teardown must be local-only — never POST server `/api/auth/logout` (a dead token 401s → re-fires the interceptor, an expiry loop). See `docs/solutions/conventions/client-401-session-expiry-gating-and-local-logout-2026-05-30.md`
- On ALL THREE teardown paths (`logout()`, `expireSession()`, `deleteAccount()`), inside one guarded (try/catch) block BEFORE the state-clearing `setState`, clear every store (so the clear always runs, never throws) in order: (1) any durable replayed-later write store FIRST — offline mutation queue, draft / pending-upload list; (2) `queryClient.clear()`; (3) token + `AsyncStorage`. If one path clears a store, all must. The query cache is keyed by query key, not user (un-cleared → cross-user read leak); the write-queue key is global with a CURRENT-bearer-token drain (un-cleared → cross-account WRITE). Clear the queue FIRST: a swallow-errors `clear*` can't throw, so a later `removeItem` reject can't skip it. A `clear*` helper with zero callers is the tell. See `docs/solutions/conventions/clear-query-cache-on-auth-teardown-2026-05-30.md` and `docs/solutions/logic-errors/durable-write-queue-not-cleared-on-auth-teardown-cross-account-replay-2026-06-19.md`
- Multi-step flows that mutate server state AND sync local state must either (a) atomically mark completion on the server step and rely on server idempotency, or (b) call `checkAuth()` / a server re-fetch in the catch when the server step succeeded but the local sync failed — never leave the user stuck behind a navigation gate after a server-side success
- The global `QueryCache.onError` net in `client/lib/query-client.ts` is scoped to **queries only** — never add a global `MutationCache.onError` toast (v5's cache-level `onError` fires in addition to each observer's, so it double-toasts local handlers). The net suppresses 4xx (incl. the `on401: "throw"` path and 429) via the `/^4\d\d:/` message guard; screens rendering their own error UI opt out with `meta: { silentError: true }`. It is the backstop — do NOT add a per-screen toast for query errors. See `docs/solutions/design-patterns/module-level-emitter-bridge-out-of-tree-to-toast-2026-05-28.md`
- `meta: { silentError: true }` is only safe on a query key when **every** consumer of that key opts out AND renders its own error UI (v5 stores `meta` on the shared cache entry — last `setOptions` wins — so a partial opt-out is **mount-order-dependent**). Before adding it, `findReferences` the **query key** (not the wrapping hook) to enumerate consumers; if any rely on the global toast, do NOT opt out — accept the toast + inline double-report. Static keys (`QUERY_KEYS.dietaryProfile`, `["/api/daily-summary"]`) are the danger; per-`id` keys rarely co-mount. See `docs/solutions/logic-errors/shared-query-key-meta-mount-order-2026-05-28.md`
- When a screen reads a query, check `isError` **before** `data.length === 0` / `!data` — a failed query defaulted to `[]` (`data ?? []` / `data || []`) otherwise slips silently into the genuine-empty branch, hiding the failure. Render an error+retry state (e.g. `EmptyState variant="temporary"` with `onAction={refetch}`) distinct from the first-run empty state. For a 404-vs-transient split, branch on `error.code === "NOT_FOUND"` (per the contract above); a query fn that throws only a bare status must be upgraded to a code-carrying `ApiError` via `throwStatusError` in `client/lib/throw-status-error.ts` so the mocked-`apiRequest` test path and production agree on `.code`. A wrapper hook must itself expose `isError`/`error`, or this check is structurally impossible for its consumers
- When a control's value derives from a fetched baseline AND a user action writes that baseline back, a failed read MUST block the write path — read `isError` and render error/retry _instead of_ the controls; don't just gate `disabled` on `isLoading`. `data?.x ?? default` masks a failed read as "empty", so toggling persists a write against a phantom baseline. See `docs/solutions/logic-errors/read-failure-phantom-baseline-write-2026-05-28.md`
- Every mutation must surface visible failure copy in `onError` — haptics-only / VoiceOver-only handlers leave other users a silent no-op; check new save/delete/log flows for a user-visible message path (toast, inline state, or equivalent), not just `onSuccess`. And wrap every `await mutateAsync(...)` in `try/catch` — `mutateAsync` rejects on failure and a hook-level `onError` doesn't suppress that rejection, so an unwrapped one is an unhandled-rejection bug
