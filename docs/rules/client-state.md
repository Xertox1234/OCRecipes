# Client State Rules

- Always use `apiRequest()` from `@/lib/query-client` for all server communication — never raw `fetch()` in components or hooks
- Always use the `Authorization` header for auth tokens — never cookies
- After an irreversible server mutation (account delete, payment, hard-delete), wrap local cleanup (`tokenStorage.clear`, `AsyncStorage.removeItem`) in try/catch and never throw — propagating a post-success error makes the user "retry" a destructive action that already succeeded
- Multi-step flows that mutate server state AND sync local state (e.g. POST profile → updateUser({onboardingCompleted: true})) must either (a) atomically mark completion on the server step and rely on server idempotency, or (b) call `checkAuth()` / equivalent server re-fetch in the catch when the server step succeeded but the local sync failed — never leave the user stuck behind a navigation gate after a server-side success
- The global `QueryCache.onError` net in `client/lib/query-client.ts` is scoped to **queries only** — never add a global `MutationCache.onError` toast. In TanStack Query v5 a cache-level `onError` fires _in addition to_ each observer's local handler, so a global mutation handler double-toasts against the existing local mutation `onError` handlers. The global query net suppresses 4xx (incl. the `on401: "throw"` path and 429) via the `/^4\d\d:/` message guard; screens that render their own error UI opt out with `meta: { silentError: true }`. See `docs/solutions/design-patterns/module-level-emitter-bridge-out-of-tree-to-toast-2026-05-28.md`.
- `meta: { silentError: true }` is only safe on a query key when **every** consumer of that key opts out AND renders its own error UI. In v5 `meta` lives on the shared Query **cache entry** (the last observer's `setOptions` wins), so opting out on a key that other screens read via the global toast makes suppression **mount-order-dependent** — it can silence their backstop (re-introducing a silent failure) or double-report. Before adding `silentError`, `findReferences` the **query key** (not just the wrapping hook) to enumerate consumers; if any rely on the global toast, do NOT opt out — accept the toast + inline double-report. Static keys (`QUERY_KEYS.dietaryProfile`, `["/api/daily-summary"]`) are the dangerous case; per-`id` keys rarely co-mount the same entry. See `docs/solutions/logic-errors/shared-query-key-meta-mount-order-2026-05-28.md`.
- When a screen reads a query, check `isError` **before** checking `data.length === 0` / `!data` — a failed query whose data is defaulted to `[]` (via `data ?? []` or `data || []`) otherwise slips silently into the genuine-empty branch, hiding the failure (and sometimes presenting confident-wrong data like a `0` count). Render an error+retry state (e.g. `EmptyState variant="temporary"` with `onAction={refetch}`) that is distinct from the legitimate first-run empty state. For a 404-vs-transient split, branch on the thrown `error.message` (the project query fns throw `new Error("<status>")`). The global `QueryCache.onError` toast is the backstop, not the per-screen affordance — do NOT add a new toast for query errors.
- When a control's value derives from a fetched baseline AND a user action writes that baseline back, a failed read MUST block the write path — read `isError` and render error/retry _instead of_ the controls, don't just gate `disabled` on `isLoading`. `data?.x ?? default` is fine for rendering but masks a failed read as "empty", so toggling persists a write against a phantom baseline that overwrites the user's real value. See `docs/solutions/logic-errors/read-failure-phantom-baseline-write-2026-05-28.md`.
