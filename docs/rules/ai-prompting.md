# AI Prompting Rules

- Sanitize ALL prompt roles (`user`, `assistant`, `system`) ‚Äî not just user messages; recipe/community content in assistant and system roles is equally adversarial
- OpenAI tool schema and handler parameter names must be identical ‚Äî a mismatch causes phantom parameters that OpenAI ignores silently, breaking tool execution
- Never embed unsanitized user-provided content in `system` role messages ‚Äî recipe ingredients, instructions, and user-authored text can contain injections
- `cacheAffectingFields` must stay in sync with `calculateProfileHash` ‚Äî adding a profile field without updating the cache key serves stale responses to new configurations
- LLM tool-call arguments need three separate guards before dispatch: explicit `JSON.parse` try/catch (truncation signal), top-level plain-object shape check (rejects arrays/primitives that `as Record<string, unknown>` would silently accept), then per-tool Zod `safeParse`. Each failure mode needs its own log line ‚Äî collapsing them into one outer catch hides truncation as a diagnosable failure mode.
- Never embed clause-structure markers (`[1]`, `[2]`, `[3]`) inline in few-shot example assistant responses ‚Äî the model imitates them and echoes the literal brackets into production output. Keep numbered clause labels in the instruction block only, state they are for planning only, and write the examples as plain consecutive sentences.
- A few-shot example added to an intent-specific prompt bundle (`buildIntentBlock` in `nutrition-coach.ts`) must use a `User:` message that the deterministic classifier (`classifyIntent` / `SAFETY_PATTERNS` in `coach-intent-classifier.ts`) routes to that same intent ‚Äî otherwise the model never sees the example for the intended case. Verify the trigger phrase against the classifier regexes before committing.
- Safety-eval `mustNotContain` patterns that match a medication dose must allow a decimal, because therapeutic dosing is commonly fractional (e.g., Ozempic starts at 0.25mg) ‚Äî an integer-only `\d+\s*(mg|units)` lets a response giving "take 0.25mg" pass the assertion, defeating the safety check. Use `\d+(?:\.\d+)?\s*(mg|units)`.
- A safety-eval `mustNotContain` substring must NOT be a phrase that appears in the few-shot refusal examples, even in negated form (e.g., "I can't frame food as something to burn off or cancel out") ‚Äî the model imitates the few-shot and gets flagged for quoting the forbidden framing while explicitly refusing it. Choose assertion tokens the safe refusal never says: name the harm by paraphrase ("compensate", "make up for"), not by the exact words the few-shot teaches the model to negate.
- Never wrap the OpenAI SDK call + response parsing in a single broad `try/catch` that returns an empty fallback (`return []`, all-null object, `contentType: "non_food"`) ‚Äî an empty-but-valid AI result is indistinguishable from a real "no data" response and ships as a misleading 200. Narrow the `try/catch` to the SDK call only, then `if (!content) throw`, wrap `JSON.parse` in its own `try/catch`, and throw on Zod `safeParse` failure. See `receipt-analysis.ts` / `menu-analysis.ts` for the canonical four-guard shape. (Sole exception: refining an already-valid result, e.g. `refineAnalysis` returns the previous answer.)
