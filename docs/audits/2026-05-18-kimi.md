## Findings

Each finding has a lifecycle: `open` -> `fixing` -> `verified` or `deferred` or `false-positive`.

**Status key:**

- `open` - Found but not yet addressed
- `fixing` - Work in progress
- `verified` - Fix applied AND confirmed by test/grep/type-check
- `deferred` - Intentionally postponed (must link to todo)
- `false-positive` - Agent was wrong or issue was already fixed

**Research key** (Phase 2.5 verdict, recorded in the `Research` column):

- `confirmed` - current documentation agrees the finding is valid
- `better-fix` - finding is real, but current docs show a cleaner fix (described in the `Verification` column for Phase 3 to use)
- `contradicted ⚠` - current docs say the flagged pattern is fine; may be a false positive - decide at triage
- `—` - research not applicable (finding does not hinge on library behavior), or finding predates Phase 2.5

### Kimi H1 False-Positive Details

After H1 was fixed, the H1 Kimi rerun produced multiple CRITICAL-labeled lines that manual review rejected:

- It claimed `pull_request_target` plus PR fetch enabled PR script execution, but the workflow checks out `github.event.pull_request.base.sha`, fetches PR head only as Git object data, and runs `bash scripts/ci-kimi-review.sh` from the base-branch working tree.
- It claimed `pip install "openai>=1.0.0,<2"` could resolve attacker-controlled PR manifests, but the workflow uses a literal package spec from the base workflow and never installs from PR files.
- It emitted two CRITICAL entries that explicitly said the current code was safe: `No issue detected here — the diff is clean on this front` and `GitHub Actions secrets are not user-overridable. This is safe.`

Manual verification: `.github/workflows/kimi-review.yml` keeps the checkout on trusted base code, `scripts/ci-kimi-review.sh` documents the no-execute invariant for fetched PR head commits, and grep confirms secrets are only passed to the trusted base-branch runner.
