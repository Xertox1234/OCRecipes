# Audit: Testing Setup

> **Date:** 2026-05-11
> **Trigger:** User-requested targeted audit of the testing setup (Vitest config, mocks, factories, coverage, test quality, pre-commit pipeline). First `testing`-scoped audit (previously a sub-domain of code-quality).
> **Domains:** code-quality (testing-specialist — discovery done in-line after agent connection failed at tool call 132)
> **Baseline:** 328 test files, ~3400+ tests passing on `main` (CI-clean, per CLAUDE.md no local self-verify); 0 type errors; 0 lint errors

## Coverage Gap Snapshot

The five highest-risk untested code paths, ranked by blast radius:

1. **`server/storage/api-keys.ts`** (182 LOC, 11 exports) — full API-key lifecycle (create, revoke, tier update, usage increment, stats), plus the verified-product nutrition cache. Security-sensitive (B2B monetization key path).
2. **`server/storage/verification.ts`** (279 LOC, 9 exports) — verified-product DB pipeline including `submitVerification`, `getUserCompositeScore`, `confirmFrontLabelData`. Business-critical: this storage layer is the foundation of the Verified Product API the project plans to sell.
3. **`server/storage/cookbooks.ts`** (253 LOC, 9 exports) — implements the documented polymorphic FK pattern (`recipeId` + `recipeType` discriminator, orphan cleanup). Documented gotcha pattern in `docs/patterns/database.md`; no test guards regressions.
4. **`server/routes/recipe-search.ts`, `recipe-catalog.ts`, `recipe-import.ts`** — three large premium-gated HTTP endpoints. Service layer has tests; the auth/validation/error-handling HTTP boundary is untested.
5. **`server/services/coach-context-builder.ts`** (96 LOC) — assembles user/profile/dietary context for the Coach Pro flow. Output drives prompt content; regressions are silent until a user notices wrong personalization.

Each finding has a lifecycle: `open` → `fixing` → `verified` or `deferred` or `false-positive`.

**Status key:**

- `open` — Found but not yet addressed
- `fixing` — Work in progress
- `verified` — Fix applied AND confirmed by test/grep/type-check
- `deferred` — Intentionally postponed (must link to todo)
- `false-positive` — Agent was wrong or issue was already fixed
