## Post-Audit Notes

- **7-domain discovery worked well.** Performance and camera reported 0 findings — camera's scope (changed files since 2026-05-11) contained no camera/scanning code, so a 0 result was expected and fast.
- **kimi-review diff-blindness.** Per-fix `kimi-review` repeatedly raised CRITICAL/WARNING findings that were false-positives because it sees only the staged diff, not full files or the schema — e.g. "`handleRouteError` not imported" (it was, line 9), "`recipeAccessGuard` may be undefined" (it is a ternary — always assigns), soft-delete `deletedAt` columns that do not exist on either recipe table, test helpers "might not accept overrides" (they do). Every such finding was verified false against the primary source before proceeding. Future audits: expect this and verify kimi findings against the full file, not the diff.
- **Out-of-scope observation — `verification_history` FK insert ordering.** While tracing `routes/verification.ts`, the data-integrity agent noted `server/storage/verification.ts` `submitVerification` inserts into `verification_history` _before_ upserting the parent `barcode_verifications` row; `verification_history.barcode` has a non-deferrable FK to `barcode_verifications.barcode`. On paper a first-ever verification of a new barcode would violate the FK at statement end. The ordering has been live since 2026-03-19 with no reported breakage. `server/storage/verification.ts` was not in this audit's changed-file scope. Tracked for a future verification-scoped audit in `todos/2026-05-16-verification-history-fk-ordering.md`.
