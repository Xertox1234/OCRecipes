1. **Bug-track classification of `apiRequest` dead code (#39).** This is the only bug-track candidate from `api.md`. Framed as `code-quality` with `severity: low` because the consequence is dead code + wrong error messages, not a crash. Reviewer may prefer to reclassify as `runtime-errors` (since the "wrong error message" symptom is a runtime issue) or keep as `code-quality`.

Kept as one file because the two halves are tightly coupled — the server cap exists _because_ the client could request unbounded sets. Reviewer may prefer to split.
3. **Tiered timeout constants live in OpenAI pattern.** The named constants (`OPENAI_TIMEOUT_FAST_MS` etc.) and the per-service-role error strategy table are kept in one file because they're meaningful together. Splitting would scatter the tiered defaults from the strategy table they belong with.

They differ in scale, not direction; the centralized version explicitly states "This supersedes the simpler `fail-fast-environment-validation` pattern for projects with many env vars." Same-direction pair, not duplicates.
5. **Atomic operations: two related patterns.** `atomic-server-endpoints-over-multi-request` (collapsing two client actions into one endpoint) and `atomic-operations-single-request` (collapsing two server steps into one transaction via a flag) cover different points on the atomicity axis. Reviewer may consider merging.
6. **`per-conversation-quota` vs `monthly-usage-cap-with-count`.** A `monthly-usage-cap-with-count` file already exists in `design-patterns/` from another batch. The per-conversation quota pattern is distinct enough (refinement-aware vs raw counter) to warrant its own file, but they cross-link conceptually. No merge candidate.
