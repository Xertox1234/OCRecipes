# Silence Xcode "ambiguous dependencies" Run Script warnings via alwaysOutOfDate

## Problem

A Run Script build phase emits:

```
⚠️  Script has ambiguous dependencies causing it to run on every build.
    To fix, go to: Xcode » … » Build Phases » '<phase name>'
```

Xcode cannot statically prove when the script needs to re-run, so it runs it
unconditionally and warns. It's cosmetic noise + wasted rebuild work, not a
failure — but for scripts that genuinely *must* run every build, the correct fix
is to **declare that intent**, not to invent fake input/output file lists.

## Symptoms

- Build log line: `Script has ambiguous dependencies causing it to run on every build.`
- The phase has empty `inputPaths`/`outputPaths` **and** no `alwaysOutOfDate` flag in `project.pbxproj`.
- The phase re-runs on every incremental build.

## Root Cause

A `PBXShellScriptBuildPhase` with no declared `inputPaths`/`outputPaths` and no
`alwaysOutOfDate = 1;` is "ambiguous" to Xcode's dependency analyzer. (Compare:
the Expo-generated `Bundle React Native code and images` and `[CP] …` phases
already set `alwaysOutOfDate = 1;` on purpose and never warn.) Setting the flag
declares "yes, run this every build deliberately" and silences the warning
without changing behavior.

## Solution

Set `alwaysOutOfDate` at every layer that can (re)materialize the phase. Where
that is depends on **who generates the phase**:

1. **The committed `project.pbxproj` is the load-bearing edit.** Add
   `alwaysOutOfDate = 1;` directly to the phase object (right after
   `isa = PBXShellScriptBuildPhase;`):

   ```
   8D0098AC… /* [MLKit] Patch for platform */ = {
       isa = PBXShellScriptBuildPhase;
       alwaysOutOfDate = 1;          /* <-- added */
       buildActionMask = 2147483647;
       …
   ```

   In a committed-native Expo project (`ios/` is git-tracked, **EAS builds the
   committed native directly with no prebuild**), this is what ships and what
   survives a routine `pod install` — see the footgun below.

2. **If the phase is generated by the Podfile `post_install` hook**, also set it
   at the source so a *regenerated-from-absence* phase still carries the flag.
   `Xcodeproj` ≥ 1.21 (confirmed on 1.27.0 / CocoaPods 1.16.2) exposes the setter:

   ```ruby
   phase = target.new_shell_script_build_phase(phase_name)
   phase.always_out_of_date = "1"   # serializes to alwaysOutOfDate = 1;
   phase.shell_script = <<~SHELL
     …
   SHELL
   ```

3. **If the phase is added by a config plugin that is NOT in `app.json`**
   (e.g. Sentry wired manually / committed-native), *nothing regenerates it* —
   not `pod install` (that only touches the pod's own `[CP]` phases), not
   prebuild (the plugin isn't registered, and prebuild is forbidden here). A
   manual `alwaysOutOfDate = 1;` edit to the committed pbxproj is fully durable.
   Verify the plugin's absence before assuming "WONTFIX": `grep sentry app.json`.

## Prevention

- **Do not delete the committed-pbxproj edit as "redundant because the Podfile
  hook sets it."** The hook is idempotent-guarded:
  `next if target.build_phases.any? { |p| p.name == phase_name }`. When the phase
  already exists (the normal case on every `pod install`), the hook **skips it
  and the `always_out_of_date` line never runs** — the committed pbxproj value is
  what carries through. A Podfile-only change is therefore *inert on a normal
  `pod install`* and the warning returns. Keep both edits.
- Verify the edits before relying on CI: there is **no iOS build in CI**, so a
  green test/type/lint gate proves nothing about this. Confirm with a local
  `pod install` + Xcode build, or trust that the declaration matches the
  Expo-generated phases that already use the flag without warning.
- `alwaysOutOfDate` is additive metadata — it does **not** change what the script
  does or when it conceptually runs (these scripts already ran every build). Net
  behavior is unchanged; only the warning disappears.
- Never `expo prebuild --clean` in a committed-native project — it wipes the
  customized `ios/` (Podfile hook + manual pbxproj edits + the whole Sentry
  wiring).
